# MentalGymnastics: Standards-Based Skill Ladder

**Status:** Programming frame  
**Applies to:** Practice design, progression design, curriculum planning, and training standards  
**Depends on:** [MentalGymnastics: Progression Against Vibes](progression-against-vibes.md)  
**Core premise:** The skill ladder defines the map. Observable standards define the gates.

## Purpose

This document defines the programming frame for MentalGymnastics before any specific progressions are designed.

It treats MentalGymnastics less like content, habit tracking, or open-ended reflection, and more like gymnastics, calisthenics, or strength training programming. The program should organize invisible mental capacities into visible advancement structures: named capacities, prerequisite relationships, constrained drills, repeatable tests, regressions, and standards for progression.

This is not yet the curriculum. It does not define the actual exercises, level sequences, or pass/fail numbers. Those come later. This document defines the shape that those progressions must fit.

## The Combined Model

The preferred structure is a **standards-based skill ladder**.

The ladder has two inseparable parts:

- **Skill tree / gymnastics levels:** the map of capacities, prerequisites, branches, and advancement paths.
- **Strict observable standards:** the gates that determine whether advancement is earned.

These are not competing models. A skill tree without standards becomes decorative progression. Standards without a tree become disconnected tests. Together, they create a discipline.

In physical training terms:

- The **skill tree** answers: what must be built before this becomes trainable?
- The **level** answers: what version of the capacity is currently being trained?
- The **standard** answers: what counts as a clean rep, hold, set, or test?
- The **programming** answers: how often, how hard, in what order, with what recovery, and when to retest?

## Foundational Commitments

The program should be designed around demonstrated capacity, not self-description.

Advancement should require evidence. The evidence does not need to be public, crude, or purely numerical, but it must be observable enough that the practitioner cannot simply reinterpret failure as progress.

The program should prefer:

- Load over exposure.
- Criteria over vibes.
- Prerequisites over enthusiasm.
- Repeatability over novelty.
- Stabilization over peak effort.
- Transfer over isolated performance.
- Regression over customization that escapes the demand.
- Testing over self-certification.

## Core Units

### Capacity

A capacity is a trainable mental ability.

It should be specific enough to load, constrain, regress, stabilize, and test. If a capacity cannot be placed under demand, it is not yet defined clearly enough for programming.

Examples of possible capacity categories may include attention, working memory, inhibition, abstraction, recall, emotional regulation, perception, judgment, or transfer. This document does not yet define the final categories.

### Drill

A drill is a constrained practice format that trains one or more capacities.

A drill is not valuable because it feels meaningful. It is valuable because it exposes a capacity to a defined demand and produces evidence about the practitioner's current limit.

### Level

A level is a named position in a progression.

It should represent a real change in demand, not a cosmetic label. A higher level should remove compensation, increase load, narrow the margin for error, add interference, require longer stabilization, or demand transfer into a less protected context.

### Standard

A standard defines what counts as successful performance.

Standards should make advancement harder to fake. They should specify the observable conditions under which a capacity must appear, such as duration, error tolerance, recovery time, consistency, interference, complexity, fatigue, or transfer.

### Gate

A gate is the advancement rule attached to a standard.

It determines whether the practitioner may move forward, repeat the current level, regress temporarily, or enter stabilization work.

### Regression

A regression is a deliberately easier version of a drill or level.

Regression is not failure. It is load management. A good regression preserves the same training principle while reducing demand enough for clean execution.

### Stabilization

Stabilization is the phase where a newly acquired capacity becomes reliable.

A capacity should not be treated as owned after a single successful test. It should survive repetition, ordinary variance, mild stress, and adjacent contexts before the ladder grants meaningful advancement.

### Transfer

Transfer is the appearance of a trained capacity outside the narrow drill that built it.

The program should eventually ask whether a capacity survives a change in format, content, pressure, or context. Without transfer, the user may only be improving at the drill.

## Programming Logic

The program should be organized like a serious physical training system.

### Prerequisites

Each branch should define what must already be stable before a new demand is introduced.

Prerequisites protect the practitioner from advancing into complexity that merely hides weakness. They also prevent the ego from skipping boring foundations in search of advanced symbols.

### Load

Load is the deliberate increase of demand on a capacity.

Load may come from duration, density, complexity, interference, ambiguity, delay, emotional pressure, precision, or transfer distance. The important point is not that the work gets harder in a vague way. The important point is that the demand increases in a named and controlled way.

### Constraints

Constraints are the rules that make the drill honest.

A constraint removes escape routes. It prevents the practitioner from compensating through speed, vagueness, excessive time, external aids, emotional interpretation, or selective memory.

### Volume

Volume is the amount of practice performed.

Volume should support adaptation, not merely produce a feeling of seriousness. More repetitions do not matter if the demand is too vague, the form is corrupted, or the work has no relationship to progression.

### Intensity

Intensity is proximity to the current limit of a capacity.

High intensity work should be used deliberately. It can reveal thresholds, but it can also create noise if the practitioner has not built stable form at lower demands.

### Frequency

Frequency is how often a capacity is exposed to training.

High-frequency work may suit basic capacities that need repeated contact. Lower-frequency work may suit tests, emotionally costly efforts, or high-interference drills. The final curriculum should define this per branch.

### Recovery

Recovery is part of programming, not a break from programming.

Mental work can degrade through fatigue, boredom, emotional saturation, or loss of precision. A serious program should distinguish between useful discomfort and degraded execution.

### Retesting

Retesting is the periodic reapplication of a standard.

Retests should confirm that a capacity remains present after time, fatigue, distraction, or adjacent training. A passed level that cannot be reproduced should not be treated as stable.

## Advancement Rules

Advancement should be earned through clean performance under the required condition.

A future progression may define exact thresholds, but the general advancement logic should follow these rules:

- The practitioner advances only when the standard is met.
- A single peak performance is weaker evidence than repeated clean performance.
- Advancement should usually require consistency across more than one session.
- Failed attempts should produce either repeated exposure, regression, or stabilization, not vague encouragement.
- The program should distinguish technical failure, effort failure, overload, and miscalibrated standards.
- The practitioner should not be allowed to customize away the primary demand of a level.

## What Counts as Observable

MentalGymnastics should not pretend that all important evidence must be external or numerical. But the evidence must constrain interpretation.

Observable evidence may include:

- Time under a defined condition.
- Number and type of errors.
- Recovery speed after loss of form.
- Accuracy after delay.
- Consistency across sessions.
- Performance under distraction or interference.
- Ability to explain, reconstruct, or apply a mental operation.
- Transfer into a related but non-identical task.
- Failure mode patterns that repeat over time.

Subjective experience may be recorded, but it should not be the primary advancement criterion.

## Anti-Self-Deception Design

The program must assume that the practitioner will be tempted to negotiate with the standard.

Common escape routes include:

- Advancing after one unusually good attempt.
- Treating effort as evidence of capacity.
- Treating insight as evidence of integration.
- Choosing novelty when fundamentals become boring.
- Reframing failure as depth.
- Avoiding retests.
- Lowering standards through customization.
- Counting participation as progress.

The ladder should make these escape routes expensive. It should preserve dignity without flattering avoidance.

## Relationship Between Branches

The final program may include multiple branches of mental capacity.

Branches should not be isolated content categories. They should describe trainable capacities and their relationships. Some branches may be prerequisites for others. Some may develop in parallel. Some may require maintenance after being unlocked.

The future progression design should define:

- Which capacities are foundational.
- Which capacities are advanced.
- Which capacities can be trained in parallel.
- Which capacities require stabilization before branching.
- Which tests prove readiness for more complex work.

## Session Shape

The final curriculum may define multiple session types, but every session should have a programming purpose.

Possible session roles include:

- **Practice:** repeated clean work below or near the current limit.
- **Load:** deliberate increase in demand.
- **Test:** formal attempt against a standard.
- **Stabilization:** consolidation of a recently passed capacity.
- **Regression:** easier work that restores clean form.
- **Transfer:** application of a capacity in a changed context.
- **Recovery:** reduced demand that preserves contact without degrading form.

A session should not exist merely to make the user feel engaged.

## Non-Goals

This document does not define:

- The final list of capacities.
- The actual skill tree.
- The level names.
- The drill library.
- The pass/fail thresholds.
- The weekly schedule.
- The user interface.
- The software implementation.

Those should be designed next, using this frame as the constraint.

## Design Test

When a future progression, drill, or programming element is proposed, it should be able to answer:

1. What capacity is being trained?
2. What is the load?
3. What constraint makes the work honest?
4. What does clean performance look like?
5. What failure modes are expected?
6. What regression preserves the same principle?
7. What standard gates advancement?
8. What proves stabilization?
9. What proves transfer?
10. How does this prevent the practitioner from mistaking participation for progress?

If those questions cannot be answered, the proposal is probably still a vibe rather than programming.
