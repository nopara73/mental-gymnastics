# MentalGymnastics: Complete Training Program

**Status:** Programming base  
**Applies to:** Curriculum design, progression logic, training standards, future domain modeling  
**Depends on:** [Progression Against Vibes](../foundation/progression-against-vibes.md), [Standards-Based Skill Ladder](../foundation/standards-based-skill-ladder.md)  
**Scope:** This is a training program document, not software design, UI design, therapy, motivation, or content marketing.

## Purpose

MentalGymnastics trains mental capacities by putting them under named load, protecting them with constraints, testing them with observable standards, and advancing only after stabilization and transfer.

The program is modeled like serious physical training:

- A **capacity** is the trainable ability.
- A **branch** is a family of related capacities.
- A **level** is a real increase in demand.
- A **drill** is a constrained training format.
- A **standard** defines clean performance.
- A **gate** decides whether the practitioner repeats, regresses, stabilizes, maintains, transfers, or advances.

Participation does not advance the practitioner. Effort does not advance the practitioner. Insight does not advance the practitioner unless it appears as repeatable performance under the required condition.

## Core Terms

### Capacity

A capacity is a trainable mental ability that can be loaded, constrained, regressed, stabilized, and tested.

### Branch

A branch is a progression path for a related set of capacities. A branch must have a reason to exist and a defined relationship to the rest of the tree.

### Level

A level is a defined demand state inside a branch. Higher levels change the demand by increasing duration, density, complexity, interference, ambiguity, delay, emotional pressure, precision, or transfer distance.

### Drill

A drill is a constrained practice format. Every drill must name its purpose, capacity, load, honesty constraint, clean performance, failure modes, and regression.

### Standard

A standard is the pass condition for a drill, level, transfer test, or maintenance check. Standards may be numerical, rubric-based, or both, but they must constrain interpretation.

### Gate

A gate is the advancement decision attached to a standard.

Gate outcomes:

- **Fail:** the standard was not met.
- **Pass once:** the standard was met once, but is not yet owned.
- **Stabilize:** repeat the standard under ordinary variance before advancement.
- **Own:** the standard is repeatable under retest conditions.
- **Maintain:** use lower volume to prevent decay.
- **Regress:** reduce load while preserving the same demand.
- **Review:** inspect branch balance, failure type, and programming fit.

### Regression

A regression is an easier variant that preserves the same principle. It may reduce duration, density, complexity, interference, ambiguity, delay, pressure, precision, or transfer distance. It must not remove the core demand.

### Stabilization

Stabilization is the phase after a pass and before meaningful advancement. A passed standard is not owned until it survives repetition, time, mild fatigue, distraction, or adjacent work.

### Transfer

Transfer is performance of the same capacity in a related but non-identical task. Transfer prevents the program from rewarding drill-specific tricks.

### Evidence Artifact

An evidence artifact is a short record that constrains interpretation: score, time, error count, reconstruction, comparison, rule explanation, failed item list, or repeatability record. Artifacts exist to support programming decisions, not to create busywork.

### BranchLevelState

A BranchLevelState is the current programming state of one branch at one level. It records whether that branch-level pair is unopened, training, test-ready, passed once, stabilizing, owned, in maintenance, or decayed.

### PractitionerState

A PractitionerState is the whole-practitioner programming picture used during placement, weekly planning, and global review. It summarizes branch-level states, maintenance currency, recent failure types, bottlenecks, volume, intensity, recovery, deload history, and current practitioner category.

## Program States

Each branch-level pair has one BranchLevelState.

| State | Meaning | Advancement Effect |
| --- | --- | --- |
| Unopened | Prerequisites are not owned. | Cannot train except as exposure during assessment. |
| Training | The level is being practiced below or near limit. | No advancement. |
| Test-ready | Recent practice meets clean-performance criteria. | Formal test may be attempted. |
| Passed once | One formal test passed. | Enter stabilizing; next level remains locked. |
| Stabilizing | Additional passes are being collected under ordinary variance, time, mild fatigue, distraction, or adjacent work. | Can become owned only after all stabilization passes are clean. |
| Owned | Pass repeated under retest conditions. | Next level may unlock if global balance allows. |
| Maintenance | Owned level receives lower-volume exposure. | Keeps later levels valid. |
| Decayed | Maintenance check failed. | Advanced work depending on it is capped until restored. |

## Global Level Ladder

All branches use the same five-level demand structure. The content differs by branch, but the meaning of each level is consistent.

| Level | Name | Real Increase in Demand | Typical Gate |
| --- | --- | --- | --- |
| L1 | Protected Control | The capacity appears in a simple, low-interference format. | Clean execution in protected drill conditions. |
| L2 | Duration and Density | The capacity survives longer exposure, more repetitions, or shorter rests. | Clean execution across repeated sets. |
| L3 | Interference and Delay | The capacity survives distraction, rule conflict, fatigue, or delayed recall. | Clean execution after adjacent work or interruption. |
| L4 | Transfer | The capacity appears in a related but non-identical task. | Transfer test passed and retested. |
| L5 | Integration Under Pressure | The capacity holds while multiple branches are active and margins narrow. | Global review plus branch test under pressure. |

L1 and L2 are foundational acquisition levels. L3 is the bridge between foundation and advanced work. L4 and L5 are advanced levels.

## Advancement Logic

Advancement is never based on participation.

1. The practitioner trains a branch level until practice sessions meet clean-performance criteria twice in a row.
2. The practitioner attempts the formal level test.
3. If the test fails, the failure is classified and the response is prescribed.
4. If the test passes once, the branch enters stabilization.
5. Stabilization requires two additional passes on different days within 14 days.
6. At least one stabilization pass must occur after mild adjacent work or a controlled distractor.
7. The level becomes owned only after all stabilization passes are clean.
8. The next level unlocks only if prerequisites and global balance rules are satisfied.

### Pass Once Versus Owned

**Passed once** means the practitioner met the standard once under formal test conditions.

**Owned** means:

- The practitioner passed the same standard three times.
- The passes occurred across at least seven calendar days.
- One pass occurred after adjacent branch work or a mild distractor.
- No pass required changing the standard.
- The practitioner can identify the main failure mode avoided during the test.

Excellence is recorded separately from ownership. Excellent performance may reduce future maintenance volume, but it does not skip levels.

### Mixed Performance

Minimum passing quality requires all critical constraints to be met. A strong score does not compensate for a broken constraint.

Mixed performance is judged in this order:

1. **Critical constraint:** if broken, the attempt fails.
2. **Output completeness:** if the artifact is incomplete, the attempt fails.
3. **Error threshold:** if errors exceed the standard, the attempt fails.
4. **Recovery:** if an error occurs but is corrected within the standard, continue.
5. **Repeatability:** if two judges or two reviews disagree on a qualitative rubric, repeat the test with a fresh but equivalent prompt.

### Passing and Excellence

**Passing** means the standard is met with no critical constraint breach.

**Excellence** means the standard is exceeded while preserving clean form. Excellence can be recorded as:

- Lower error count than required.
- Faster recovery after drift or interruption.
- Clearer reconstruction.
- Correct explanation of method and failure prevention.
- Successful transfer at a longer distance than required.

Excellence never bypasses prerequisites.

## Branch Map

The program has five foundational branches and three advanced branches.

| Branch | Type | Reason to Exist | Relationship to Tree | Unlock Rule |
| --- | --- | --- | --- | --- |
| FH: Focus Hold | Foundational | Trains holding a selected target without drifting into vague exposure. | Prerequisite for every branch; maintenance branch after L2. | Universal start. |
| FS: Focus Shift and Recovery | Foundational | Trains deliberate switching and return instead of accidental attention drift. | Parallel branch after FH L1; prerequisite for transfer and integration. | FH L1 passed once. |
| WM: Working Memory and Reconstruction | Foundational | Trains holding, manipulating, and reproducing information without external support. | Parallel branch; prerequisite for Concept Operations and Transfer Integration. | FH L1 passed once. |
| IR: Inhibition and Response Control | Foundational | Trains withholding automatic responses and obeying task rules under pressure. | Prerequisite for advanced branches; protects standards from cheating. | FH L1 passed once. |
| DE: Discrimination and Error Checking | Foundational | Trains detecting distinctions, comparing outputs, and finding errors. | Parallel branch; prerequisite for Concept Operations and global reviews. | FH L1 passed once. |
| CO: Concept Operations | Advanced | Trains rule extraction, abstraction, mapping, and controlled inference. | Advanced branch; depends on WM, IR, and DE. | WM L3, IR L3, and DE L3 owned. |
| AI: Affective Interference Control | Advanced | Trains preserving task standards under frustration, uncertainty, time pressure, or evaluative pressure. | Advanced branch; depends on FH, FS, and IR; transfer branch for pressure. | FH L3, FS L3, and IR L3 owned. |
| TI: Transfer Integration | Advanced | Trains combined use of capacities in less protected contexts. | Transfer branch and global integration branch. | All foundational L3 owned; at least one CO L2 or AI L2 owned. |

No branch may advance more than two owned levels ahead of the lowest foundational branch. Advanced branches pause if any prerequisite branch decays below maintenance standard.

## Trainable Capacities

| Capacity | Branch | What Is Trained | How It Is Loaded | How It Is Constrained | How It Is Tested |
| --- | --- | --- | --- | --- | --- |
| Selective hold | FH | Keep a selected object or operation in the foreground. | Duration, distractors, precision of return. | No switching target; drift must be marked. | Timed hold with drift count and recovery time. |
| Return after drift | FH, FS | Notice loss of target and resume without resetting the session. | Shorter recovery window, more interruptions. | Drift must be marked immediately. | Return-time standard after controlled interruption. |
| Deliberate switching | FS | Move attention between specified targets on command. | Switch count, rule complexity, interval density. | Switches occur only on cue; no anticipatory switching. | Alternating target test with sequence accuracy. |
| Encoding fidelity | WM | Take in information accurately enough to reconstruct it later. | Item count, detail density, delay. | No rereading after encode window. | Delayed reconstruction and comparison. |
| Manipulation in mind | WM | Transform held information without externalizing the intermediate steps. | Operation steps, reversals, interference. | Intermediate notes prohibited unless drill allows them. | Final reconstruction plus operation explanation. |
| Response inhibition | IR | Withhold a tempting or automatic response. | Cue conflict, speed pressure, emotional pressure. | Premature response fails the item. | Go/no-go, stop, or rule-conflict test. |
| Rule fidelity | IR, CO | Follow the assigned rule even when another rule is easier. | Rule count, exceptions, switching. | Rule must be stated before the set. | Error count and explanation after set. |
| Fine discrimination | DE | Detect relevant differences and ignore irrelevant ones. | Similarity, quantity, time limit. | Guessing must be marked; unmarked guesses fail. | Comparison task with false-positive and false-negative counts. |
| Error audit | DE | Find errors in one's own output. | Output length, delay, error subtlety. | Original output cannot be changed during audit. | Audit report compared with seeded or known errors. |
| Rule extraction | CO | Infer the rule behind examples without overfitting. | Ambiguity, negative examples, transfer distance. | Must state rule before new examples. | Rule tested on unseen examples. |
| Abstraction mapping | CO | Map a structure from one context to another. | Distance between contexts, number of relations. | Mapping must preserve relations, not surface terms. | New-context application scored by relation accuracy. |
| Pressure-stable execution | AI | Preserve the same standard under frustration, uncertainty, or evaluation. | Time limit, consequence simulation, ambiguity, observation. | Standard cannot be lowered during pressure condition. | Branch test repeated under defined pressure. |
| Recovery after disruption | AI, TI | Resume clean execution after interruption or mistake. | Interruption timing, restart delay, task complexity. | No full restart unless standard allows it. | Post-disruption performance compared with pre-disruption baseline. |
| Integrated task control | TI | Combine multiple capacities without hiding a weak branch. | Branch count, task length, transfer distance. | Each component branch must leave evidence. | Composite task with branch-specific scoring. |

## Load Rules by Branch

Load increases must be coherent. Only one primary load variable should increase at a time during acquisition. Two variables may increase during advanced integration only when both are already stable separately.

| Branch | Primary Load Variables | Prohibited Load Increase |
| --- | --- | --- |
| FH | Duration, distractor salience, recovery window, target subtlety. | Adding complex reasoning before hold is stable. |
| FS | Switch count, cue density, rule contrast, return precision. | Random switching without a defined cue rule. |
| WM | Item count, detail density, operation steps, delay, interference. | Adding abstraction when encoding is inaccurate. |
| IR | Cue conflict, response speed, exception count, pressure. | Increasing speed after rule-breaking errors. |
| DE | Similarity of items, quantity, error subtlety, audit delay. | Increasing quantity when false positives are high. |
| CO | Rule ambiguity, example count, exception handling, transfer distance. | Ambiguity before the practitioner can state a testable rule. |
| AI | Time pressure, evaluative pressure, frustration, uncertainty. | Emotional pressure that prevents clean evidence collection. |
| TI | Number of branches, task length, domain distance, interference. | Novel task format without branch-specific scoring. |

## Volume, Intensity, Frequency, and Recovery

### Volume

Volume must support adaptation. It is measured as clean exposure to a capacity, not minutes spent near the task.

Default acquisition volume:

- Foundational branch: 2 to 4 working sets per exposure.
- Advanced branch: 1 to 3 working sets per exposure.
- Working set: 3 to 10 minutes, or a fixed item set when time is the wrong unit.
- Stop a set when form degrades beyond the drill's error threshold.

Volume increases only when the previous volume produced mostly clean work. Adding volume to corrupted form is bad programming.

### Intensity

Intensity means proximity to the practitioner's current limit.

| Intensity | Meaning | Use |
| --- | --- | --- |
| Low | Clean execution with obvious reserve. | Warm-up, recovery, maintenance, regression. |
| Moderate | Clean execution with occasional corrected errors. | Most practice and stabilization. |
| High | Near the current standard; errors likely if form slips. | Load sessions and formal tests. |

High intensity is not emotional drama. A quiet task at the edge of recall accuracy can be high intensity. A stressful task with sloppy criteria is not valid intensity.

### Frequency

| Branch Group | Frequency During Acquisition | Frequency During Maintenance |
| --- | --- | --- |
| FH, FS | 3 to 5 exposures per week. | 1 to 2 checks per week. |
| WM, IR, DE | 2 to 4 exposures per week. | 1 check per week. |
| CO | 2 to 3 exposures per week. | 1 check every 7 to 10 days. |
| AI | 1 to 2 exposures per week. | 1 check every 10 to 14 days. |
| TI | 1 to 2 exposures per week. | 1 global task every 14 days. |

### Recovery and Deload

Recovery protects clean execution. It is not an excuse to avoid the branch.

Use a recovery session when:

- Two consecutive sets fail from overload.
- Error count rises while load is unchanged.
- The practitioner breaks honesty constraints.
- Adjacent branches show broad decay.
- The same branch was tested at high intensity within the last 24 hours.

Recovery session rules:

- Reduce one load variable by one level.
- Keep the core constraint.
- Record one evidence artifact only.
- Do not test advancement.

Use a deload week when two or more branches show overload or decay in the same week. During deload, total working sets drop by roughly one third, tests are suspended, and maintenance checks remain.

## Session Types

| Session Type | Purpose | Typical Intensity | Evidence | Advancement Effect |
| --- | --- | --- | --- | --- |
| Practice | Build clean reps below or near limit. | Low to moderate. | One set record or output sample. | Makes test readiness possible. |
| Load | Increase one named demand variable. | Moderate to high. | Load variable, errors, failure mode. | May qualify for test readiness. |
| Test | Attempt a formal standard. | High. | Full score and artifact. | Can produce pass once. |
| Stabilization | Repeat a passed standard under ordinary variance. | Moderate to high. | Repeatability record. | Can convert pass once to owned. |
| Regression | Restore clean form at lower load. | Low to moderate. | Failure mode and restored standard. | Can return to training state. |
| Transfer | Test capacity in related but non-identical task. | Moderate to high. | Transfer artifact plus branch mapping. | Required for L4 and TI. |
| Recovery | Preserve contact while reducing fatigue or noise. | Low. | Minimal artifact. | No advancement. |

## Starting and Placement

All practitioners begin with the universal assessment. Placement must not flatter the practitioner into advanced work.

### Universal Assessment

The assessment tests FH, FS, WM, IR, and DE at L1. CO, AI, and TI are not available during initial placement.

Initial placement outcomes:

- Fail FH L1: start with FH L1 regression and no other branch advancement.
- Pass FH L1 only: begin FH L1 stabilization and open FS, WM, IR, and DE at L1 practice.
- Pass multiple foundational L1 tests: begin those branches in stabilization.
- Pass all foundational L1 tests: begin L2 training, not L3.
- Exceptional performance: may record excellence, but initial placement may not exceed L2 training.

The first two weeks are provisional. A branch is not placed above L1 unless the practitioner repeats the standard under ordinary weekly fatigue.

### Practitioner Categories

| Category | Demonstrated-Capacity Definition |
| --- | --- |
| Beginner | Any foundational branch below L2 owned, or no advanced branch unlocked. |
| Intermediate | All foundational branches L3 owned, maintenance checks current, and at least one advanced branch opened. |
| Advanced | All foundational branches L4 owned or maintained, CO and AI at least L3 owned, TI at least L3 owned, and the last global review passed. |

These categories are programming states, not identities.

## Branch Progressions

Each branch has level standards, gates, maintenance logic, transfer rules, and bottleneck implications.

### FH: Focus Hold

**Capacity:** Selective hold and return after drift.  
**Branch relationship:** Foundational prerequisite for all branches; maintenance branch after L2.  
**Primary evidence:** Timed hold, drift marks, recovery time.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Hold one simple target for 3 minutes. | No more than 5 marked drifts; each return within 10 seconds; no target change. | Pass once enters stabilization. | Repeat twice within 14 days; one after a short WM set. | Hold a different target type with same standard. |
| L2 | Hold for 6 minutes or two 3-minute sets with 60 seconds rest. | No more than 7 total drifts; average return within 8 seconds. | Own L1 required. | Repeat on two days; one after light distraction. | Hold during low-noise environment change. |
| L3 | Hold with controlled distractor. | 5 minutes with periodic irrelevant prompts; no response to distractor; no more than 5 drifts. | Own L2 required. | Repeat after FS practice. | Hold while using unfamiliar but simple content. |
| L4 | Transfer hold to another branch task. | Maintain stated target while completing WM or DE task; branch score remains passing. | Own L3 required. | Two transfer passes in different branches. | Required transfer is the level test. |
| L5 | Hold under integrated pressure. | Maintain target during 12-minute TI task; no branch-critical constraint breach. | Own L4 and TI L3 required. | Repeat during global review cycle. | Global task with branch-specific scoring. |

**Maintenance:** After L2 owned, perform one 3-minute hold check weekly. After L4 owned, perform one transfer hold every 14 days.  
**Bottleneck effect:** If FH maintenance fails, no branch may test above current level until FH is restored.  
**Regression:** Shorten duration by half while keeping drift marking and return standards.

### FS: Focus Shift and Recovery

**Capacity:** Deliberate switching, cue obedience, and clean return.  
**Branch relationship:** Foundational parallel branch; prerequisite for AI and TI.  
**Primary evidence:** Sequence accuracy, premature switches, return time.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Alternate between two targets on cue for 4 minutes. | At least 90% correct cue responses; no more than 3 anticipatory switches. | FH L1 passed once. | Repeat twice; one after FH hold. | Use a new pair of targets. |
| L2 | Increase cue density. | 6 minutes; at least 92% correct; recovery after wrong switch within next cue. | Own L1 required. | Two passes on separate days. | Switch between task types, not just objects. |
| L3 | Add rule conflict. | Switch only on valid cues while ignoring invalid cues; at least 90% valid response and 90% invalid inhibition. | Own L2 and IR L2 required. | Repeat after WM set. | Transfer to DE comparison task. |
| L4 | Transfer switching to complex work. | Alternate between two branch tasks without losing either task's passing standard. | Own L3 required. | Two different branch pairings. | Required transfer is the level test. |
| L5 | Integrated switching under pressure. | Maintain rule fidelity through 15-minute TI task with scheduled and unscheduled switches. | Own L4 and TI L3 required. | Repeat in global review. | Global task with switch log. |

**Maintenance:** One cue-switch check weekly; one transfer switch check every 14 days after L4.  
**Bottleneck effect:** FS failure limits AI and TI because the practitioner cannot prove pressure recovery or task transition.  
**Regression:** Reduce to two targets, longer cue intervals, and no invalid cues.

### WM: Working Memory and Reconstruction

**Capacity:** Encoding fidelity, mental manipulation, delayed reconstruction.  
**Branch relationship:** Foundational parallel branch; prerequisite for CO and TI.  
**Primary evidence:** Reconstruction accuracy, detail omissions, operation errors.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Encode and reconstruct 5 simple items after 60 seconds. | At least 4 of 5 exact; no invented items. | FH L1 passed once. | Repeat twice with new item sets. | Use a different content type. |
| L2 | Increase item count or detail density. | 7 items or equivalent detail set after 90 seconds; at least 85% accurate. | Own L1 required. | Two passes; one after FH or FS. | Reconstruct from visual and verbal formats. |
| L3 | Add manipulation and delay. | Transform 6 held items by a stated rule after 2-minute delay; at least 80% accurate; rule explanation correct. | Own L2 and IR L2 required. | Repeat with different rule families. | Apply memory operation inside DE task. |
| L4 | Transfer to untrained content. | Reconstruct structure, not surface wording, after 5-minute delay in a new domain; at least passing rubric. | Own L3 required. | Two transfer contexts. | Required transfer is the level test. |
| L5 | Integrated memory under pressure. | Preserve critical information through 15-minute TI task; no critical omissions. | Own L4 and TI L3 required. | Repeat during global review. | Global task with delayed reconstruction. |

**Maintenance:** One delayed reconstruction weekly; after L4, one cross-format reconstruction every 14 days.  
**Bottleneck effect:** WM errors block CO advancement because abstraction cannot be trusted if source material is not held accurately.  
**Regression:** Reduce item count or delay while preserving no-reread and no-invention constraints.

### IR: Inhibition and Response Control

**Capacity:** Response inhibition, rule fidelity, and impulse control under task constraints.  
**Branch relationship:** Foundational branch; prerequisite for CO, AI, and TI; protects anti-cheating logic.  
**Primary evidence:** Premature responses, rule violations, corrected errors.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Follow a simple go/no-go rule. | At least 90% correct; no more than 2 premature responses. | FH L1 passed once. | Repeat with different cue set. | Use a non-identical cue format. |
| L2 | Add speed or exception pressure. | At least 90% correct under fixed pace; all exceptions named before set. | Own L1 required. | Two passes; one after FS. | Use rule conflict in WM or DE. |
| L3 | Withhold easier rule under conflict. | At least 88% correct; no unmarked rule drift; correction within 2 items after error. | Own L2 required. | Repeat with different conflict rule. | Transfer to CO example sorting. |
| L4 | Transfer inhibition to open task. | Maintain stated rule while solving branch task; no critical rule breach. | Own L3 required. | Two open-task transfers. | Required transfer is the level test. |
| L5 | Inhibition under integrated pressure. | Preserve rule fidelity through TI task with interruption and time pressure; no critical breach. | Own L4 and TI L3 required. | Repeat during global review. | Global task with rule audit. |

**Maintenance:** One rule-conflict check weekly; after L4, one open-task rule audit every 14 days.  
**Bottleneck effect:** IR failure invalidates tests where the practitioner could have changed rules mid-attempt.  
**Regression:** Remove speed pressure or reduce exception count while preserving the rule and no-premature-response standard.

### DE: Discrimination and Error Checking

**Capacity:** Fine discrimination, comparison, and audit of output.  
**Branch relationship:** Foundational branch; prerequisite for CO and global reviews.  
**Primary evidence:** False positives, false negatives, seeded-error detection.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Compare simple pairs for relevant differences. | At least 90% accuracy; guesses marked; no more than 2 unmarked guesses. | FH L1 passed once. | Repeat with new pair set. | Use another item type. |
| L2 | Increase similarity and quantity. | At least 88% accuracy across 20 comparisons; false positives and false negatives both below threshold. | Own L1 required. | Two passes; one after WM. | Compare outputs produced by self. |
| L3 | Audit delayed output. | Find at least 80% of seeded errors after 5-minute delay; no more than 2 false corrections. | Own L2 and WM L2 required. | Repeat with different error types. | Audit WM or CO artifact. |
| L4 | Transfer audit to unstructured output. | Identify critical errors and uncertainty in a short self-produced artifact. | Own L3 required. | Two transfer artifacts. | Required transfer is the level test. |
| L5 | Integrated audit under pressure. | Audit TI artifact under time limit; find all critical errors and at least 80% noncritical errors. | Own L4 and TI L3 required. | Repeat during global review. | Global task audit. |

**Maintenance:** One comparison or audit check weekly; after L4, one unstructured audit every 14 days.  
**Bottleneck effect:** DE failure makes global reviews unreliable because the practitioner cannot detect weak evidence.  
**Regression:** Use fewer items or larger differences while preserving marked-guess and comparison constraints.

### CO: Concept Operations

**Capacity:** Rule extraction, abstraction mapping, controlled inference.  
**Branch relationship:** Advanced branch; depends on WM, IR, and DE.  
**Primary evidence:** Stated rule, unseen-example performance, mapping accuracy.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Extract rule from clear examples. | State a testable rule; classify unseen examples at least 85% correctly. | WM L3, IR L3, DE L3 owned. | Repeat with two rule families. | Use a different content domain. |
| L2 | Add exceptions and negative examples. | Rule handles exceptions without changing after feedback; unseen classification at least 82%. | Own L1 required. | Two passes; one after DE audit. | Explain failed examples without rewriting history. |
| L3 | Map structure across domains. | Preserve at least 80% of required relations; no surface-only mapping accepted. | Own L2 required. | Repeat with two distant domains. | Apply mapping to a new problem. |
| L4 | Transfer abstraction to open problem. | Produce a rule or model that predicts unseen cases and survives audit. | Own L3 and DE L4 required. | Two open-problem transfers. | Required transfer is the level test. |
| L5 | Concept operation under pressure. | Build and audit a model inside TI task; critical assumptions named; prediction test passed. | Own L4 and TI L3 required. | Repeat during global review. | Global task model audit. |

**Maintenance:** One rule extraction or mapping check every 7 to 10 days; after L4, one open-problem audit every 14 days.  
**Bottleneck effect:** CO stalls route first to WM, IR, or DE depending on whether the failure was memory, rule drift, or audit weakness.  
**Regression:** Use clearer examples or fewer relations while preserving pre-stated rule and unseen-example testing.

### AI: Affective Interference Control

**Capacity:** Pressure-stable execution and recovery after disruption.  
**Branch relationship:** Advanced branch; depends on FH, FS, and IR; pressure-transfer branch.  
**Primary evidence:** Branch score under pressure, rule breaches, recovery after disruption.

AI does not train the practitioner to feel calm. It trains preservation of task standards when frustration, uncertainty, time pressure, or evaluation would normally degrade execution.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Repeat an owned foundational standard under mild time pressure. | Original branch standard remains passing; no critical constraint breach. | FH L3, FS L3, IR L3 owned. | Repeat with two foundational branches. | Use a different pressure source. |
| L2 | Add uncertainty or evaluative pressure. | Standard remains passing; uncertainty is marked rather than hidden. | Own L1 required. | Two passes; one after a failed practice item. | Transfer to WM or DE. |
| L3 | Recover after controlled disruption. | Resume within defined window and finish above branch pass threshold. | Own L2 required. | Repeat with different disruption timing. | Transfer to CO or TI pretask. |
| L4 | Transfer pressure control to open task. | Complete open branch task without lowering stated standard. | Own L3 and relevant branch L4 required. | Two open-task transfers. | Required transfer is the level test. |
| L5 | Integrated pressure tolerance. | Complete TI pressure task; each component branch remains at passing quality. | Own L4 and TI L3 required. | Repeat during global review. | Global pressure task. |

**Maintenance:** One pressure repeat every 10 to 14 days; avoid stacking AI tests on consecutive days.  
**Bottleneck effect:** AI failure does not erase the underlying branch level, but it blocks pressure-based advancement and TI pressure tests.  
**Regression:** Reduce pressure source while preserving the same branch standard.

### TI: Transfer Integration

**Capacity:** Integrated task control across branches.  
**Branch relationship:** Advanced transfer branch; evaluates whether earlier capacities survive combined work.  
**Primary evidence:** Composite artifact with branch-specific scoring.

| Level | Demand | Standard | Gate | Stabilization | Transfer |
| --- | --- | --- | --- | --- | --- |
| L1 | Combine two foundational branches. | Both branch standards remain passing in one task; evidence separated by branch. | All foundational L3 owned; CO L2 or AI L2 owned. | Repeat with two branch pairings. | New task format with same pair. |
| L2 | Combine three branches with delay. | All component branches meet minimum passing quality; delayed artifact included. | Own L1 required. | Two passes; one after maintenance work. | New content domain. |
| L3 | Add advanced branch. | Foundational branch scores remain passing while CO or AI demand is active. | Own L2 and advanced branch L2 owned. | Repeat with different advanced branch or pressure source. | New task family. |
| L4 | Transfer to untrained but related problem. | Branch evidence remains visible; no bottleneck branch falls below pass. | Own L3 required. | Two far-transfer tasks. | Required transfer is the level test. |
| L5 | Global performance review task. | Multi-branch task passes, audit passes, delayed reconstruction passes, and pressure rule remains intact. | Own L4, CO L4, AI L4 required. | Repeat in separate global review cycles. | Global review itself. |

**Maintenance:** One composite task every 14 days after L1 owned; global task every 4 weeks after L3 owned.  
**Bottleneck effect:** A TI failure routes back to the weakest scored branch; TI cannot hide weak fundamentals.  
**Regression:** Reduce branch count while preserving branch-specific evidence.

## Drill Library

These drills are the standard building blocks. Branch levels may use variants, but variants must preserve the named purpose and honesty constraint.

### Drill Table

| --- | --- | --- | --- | --- | --- | --- | --- |
| FH-1 Target Hold | Build selective hold. | Selective hold, return after drift. | Duration, target subtlety, distractor salience. | Target is stated before set; every drift is marked. | Target maintained within drift threshold; returns inside time window. | Unmarked drift, target substitution, restarting after drift. | Shorter duration with same marking rule. |

| FS-2 Invalid Cue Filter | Switch while ignoring invalid cues. | Switching plus inhibition. | Invalid cue ratio, rule contrast. | Invalid cues must not trigger switch. | Valid and invalid response thresholds both met. | Treating all cues as valid, freezing, rule drift. | Reduce invalid cue count. |

## Testing Structure

### Test Readiness

A practitioner may test a level only when:

- The prerequisite level is owned.
- The same drill demand was clean in two recent practice sessions.
- No prerequisite maintenance check is overdue.
- The practitioner can state the standard before the test.
- The practitioner can name the honesty constraint.

### Assigned Work Before Tests

No graduation test may be attempted until the work that prepares its demand has been completed cleanly. This prevents a standard from asking for a capacity the program has not trained.

| Target Test | Required Preparation Before Test | What This Prepares |
| --- | --- | --- |
| L1 protected control | Two practice exposures using the branch's primary L1 drill; regression used first if clean form is not available. | Basic form, honesty constraint, minimum evidence artifact. |
| L2 duration and density | L1 owned; two load sessions increasing only duration, item count, cue density, or set count; one recovery or low-load exposure if errors rise. | Longer clean exposure without changing the core task. |
| L3 interference and delay | L2 owned; two practice exposures with the relevant interference, delay, conflict, or audit condition; one adjacent branch exposure before the final practice attempt. | Survival of the capacity after distraction, delay, or adjacent work. |
| L4 transfer | L3 owned; one near-transfer practice and one different near-transfer practice; source branch standard remains visible in both. | Applying the capacity outside the original drill without losing the original demand. |
| L5 integration under pressure | L4 owned; TI prerequisite met; one composite practice, one audit or reconstruction artifact, and one recovery exposure within the same review cycle. | Combined work without hiding a weak component branch. |

Branch-specific preparation:

- FH tests use FH-1 before FH-2, then transfer hold inside WM or DE.
- FS tests use FS-1 before FS-2, then switch between branch tasks.
- WM tests use WM-1 before WM-2, then delayed reconstruction inside DE or TI.
- IR tests use IR-1 before IR-2, then rule fidelity inside CO or TI.
- DE tests use DE-1 before DE-2, then audit of self-produced artifacts.
- CO tests use CO-1 before CO-2, then open problem transfer.
- AI tests use the already-owned source branch standard before AI-1 or AI-2.
- TI tests use TI-1 before TI-2, and every component branch must leave evidence.

### Formal Test

A formal test records:

- Branch and level.
- Drill or transfer task.
- Load variables.
- Standard.
- Critical constraints.
- Score or rubric result.
- Failure mode if failed.
- Artifact location or summary.
- Pass state: fail, pass once, stabilization pass, owned, or maintenance pass.

### Retesting Rules

Retesting checks whether the capacity survives time, fatigue, distraction, or adjacent work.

Retest requirements:

- L1 and L2 ownership require three passes across at least seven days.
- L3 ownership requires one pass after adjacent branch work or controlled distraction.
- L4 ownership requires two transfer contexts.
- L5 ownership requires two global review cycles.
- Maintenance failures require restoration before dependent advancement resumes.

Retests must not use identical prompts unless the capacity is timing or inhibition under known cues. For memory, discrimination, concept, and transfer work, use equivalent but fresh content.

### Transfer Tests

Transfer tests must be related to the trained capacity but not identical to the drill.

| Source Branch | Transfer Test | What Must Stay the Same | What Must Change |
| --- | --- | --- | --- |
| FH | Hold target during WM or DE task. | Target, drift marking, return standard. | Content and task format. |
| FS | Switch between two branch tasks. | Cue obedience and return standard. | Target type and branch context. |
| WM | Reconstruct structure from unfamiliar content. | Encoding, delay, no invention. | Domain or representation. |
| IR | Maintain rule in open task. | Pre-stated rule and no critical breach. | Task format and temptation source. |
| DE | Audit unstructured output. | Marked uncertainty and error comparison. | Output type and error distribution. |
| CO | Apply rule to unseen problem. | Testable rule and prediction check. | Domain or example set. |
| AI | Repeat standard under new pressure source. | Original branch standard. | Pressure source. |
| TI | Solve new composite task. | Branch-specific evidence. | Task family or transfer distance. |

Novelty is not advancement. A task counts as transfer only when the trained capacity is specified and the branch standard remains visible.

## Failure Classification and Response

Failure is useful only when classified.

| Failure Type | Definition | Evidence | Response |
| --- | --- | --- | --- |
| Technical failure | The practitioner does not understand or execute the form of the drill. | Confusion about rule, missing artifact, wrong procedure. | Stop test; return to practice; simplify instruction; retest later. |
| Effort failure | The practitioner avoids the standard, rushes, skips marking, or changes the rule. | Broken honesty constraint, missing drift marks, unmarked guesses. | Fail attempt; repeat at same or lower load with stricter evidence. |
| Overload | The demand exceeds current capacity despite honest execution. | Errors rise after load increase; constraint preserved. | Reduce one load variable; train regression; retest after clean practice. |
| Bad programming | The plan produced failure through poor sequencing, volume, recovery, or prerequisite mismatch. | Multiple branches degrade, repeated overload, prerequisite not stable. | Deload; restore prerequisites; revise weekly emphasis; no new tests. |

### What Happens After Failure

- First technical failure: practice the drill form at low intensity before another test.
- First effort failure: repeat the same level only after a clean practice artifact.
- First overload failure: regress one load variable and stabilize.
- Repeated overload in same branch: inspect prerequisite branch and reduce weekly load.
- Bad programming: suspend advancement testing for one week and run maintenance checks.

Failed sessions are not rebranded as success. They produce useful information when the failure type, failed constraint, and next programming action are recorded.

## Stuck State

A practitioner is stuck when ordinary repeat, regress, and stabilize responses are not restoring progress.

Stuck criteria:

- The same branch gate fails three times across at least 10 calendar days.
- The same critical constraint fails in two consecutive regression sessions.
- A prerequisite branch repeatedly decays while the practitioner trains a dependent branch.
- Two global reviews in a row name the same bottleneck without improvement in evidence.
- The practitioner can pass isolated drills but fails two related transfer tests in the same branch.

Stuck response:

- Stop advancement tests in the stuck branch.
- Identify whether the stuck point is technical failure, effort failure, overload, bad programming, or prerequisite decay.
- Train the nearest prerequisite branch for one week at moderate intensity.
- Use one regression that preserves the failed constraint.
- Reduce total weekly load if failures appear in more than one branch.
- Retest only the failed constraint before retesting the whole gate.
- If transfer is the stuck point, return to the source branch and test a nearer transfer distance.

A stuck state is resolved only when the failed constraint passes twice on different days or the relevant prerequisite maintenance check is restored.

## Regressions

A regression must preserve the principle.

Allowed regression moves:

- Shorter duration.
- Fewer items.
- Lower cue density.
- Longer rest.
- Clearer examples.
- Less subtle errors.
- Shorter delay.
- Milder pressure.
- Fewer branches in composite task.

Forbidden regression moves:

- Removing drift marking from FH.
- Allowing target changes during FH.
- Allowing uncued switches during FS.
- Allowing rereading during WM after the encode window.
- Allowing premature responses in IR.
- Allowing unmarked guesses in DE.
- Allowing vague rules in CO.
- Lowering the original branch standard during AI.
- Removing branch-specific evidence during TI.

If the forbidden move is needed for completion, the practitioner is not regressing; they are doing a different task that does not count for this branch.

## Maintenance

Maintenance is lighter than acquisition but strong enough to prevent decay.

Maintenance rules:

- Owned L1-L2 foundational branches: one check weekly.
- Owned L3 foundational branches: one standard or transfer check every 7 to 10 days.
- Owned advanced branches: one check every 10 to 14 days.
- TI L3 or higher: one global composite every 4 weeks.

Maintenance pass:

- Standard is met at one level below the owned level or at the owned level with reduced volume.
- Critical constraints remain intact.
- Evidence artifact is short but sufficient.

Maintenance fail:

- One failed check creates a warning.
- Two failed checks in a branch mark the branch decayed.
- Decayed prerequisite branches cap dependent branches at maintenance only.
- Restore by passing the last owned standard once and one lower-load transfer check.

## Global Balance and Bottlenecks

The program prevents advancement through strengths while ignoring blockers.

### Global Balance Rules

- No foundational branch may be more than two owned levels behind another foundational branch.
- No advanced branch may advance if any prerequisite branch has an overdue maintenance check.
- TI cannot advance if any component branch scored below passing in the last global review.
- CO cannot advance if WM or DE is decayed.
- AI cannot advance if FH, FS, or IR is decayed.
- A practitioner cannot be classified as advanced unless the last global review passed.

### Bottleneck Capacities

| Bottleneck | How It Shows Up | Programming Effect |
| --- | --- | --- |
| FH return after drift | Many tasks begin clean but dissolve. | Reduce task length and train FH before adding complexity. |
| WM encoding fidelity | Later reasoning uses wrong source material. | Pause CO and TI; train WM reconstruction. |
| IR rule fidelity | Standards are changed mid-task. | Suspend tests that rely on self-enforced rules. |
| DE audit accuracy | Practitioner cannot find errors in artifacts. | Require DE maintenance before global review. |
| FS recovery | Transitions cause loss of both tasks. | Reduce composite tasks; train switch recovery. |
| AI pressure stability | Performance collapses only under pressure. | Keep branch level but block pressure transfer until AI restored. |

Bottlenecks influence programming decisions. They are not character judgments.

## Global Review

A global review occurs every 4 weeks for intermediate and advanced practitioners, and every 6 weeks for beginners.

The review evaluates the whole practitioner, not isolated drill performance.

Review inputs:

- Current owned level by branch.
- Maintenance status.
- Last three failures and classifications.
- Evidence artifacts from tests and transfer tasks.
- Bottleneck branch.
- Volume and intensity history.
- Recovery or deload history.

Review decisions:

- Continue current progression.
- Emphasize bottleneck branch.
- Restore decayed branch.
- Open an advanced branch.
- Pause tests for deload.
- Attempt TI transfer.

Global review pass requires:

- No prerequisite branch decayed.
- No overdue maintenance check.
- Bottleneck branch has a programmed response.
- At least one transfer or stabilization artifact is current.
- No branch advanced by participation alone.

## Customization Rules

Customization may personalize content but not bypass demand.

Allowed:

- Content domain, as long as standard and capacity stay the same.
- Session day, as long as frequency and recovery rules remain.
- Equivalent prompt set.
- Pressure source for AI, as long as it is defined before the test.
- Transfer domain, as long as transfer distance is stated.

Not allowed:

- Skipping prerequisites.
- Replacing tests with self-description.
- Changing pass thresholds after starting.
- Removing evidence artifacts.
- Using novelty as advancement.
- Choosing only strong branches while bottlenecks decay.

## Evidence Artifacts

Artifacts should be minimal and useful.

Required for every formal test:

- Branch, level, date.
- Drill or transfer task.
- Load variables.
- Standard and critical constraint.
- Score, rubric result, or pass/fail.
- Failure classification if failed.
- Short output sample or reference.

Required for ordinary practice:

- Only one artifact per session: best set, failed set, or bottleneck note.

Artifacts are busywork if they do not affect programming. If an artifact will not change advancement, regression, maintenance, retesting, or bottleneck decisions, do not require it.

## Qualitative Rubrics

Some capacities need qualitative judgment. Qualitative does not mean vague.

### Rule Quality Rubric

| Rating | Observable Meaning |
| --- | --- |
| Fail | Rule is circular, motivational, unfalsifiable, or changed after feedback. |
| Pass | Rule is stated before test, predicts unseen cases, and handles defined exceptions. |
| Excellent | Rule is concise, predicts unseen cases, names limits, and survives audit. |

### Mapping Quality Rubric

| Rating | Observable Meaning |
| --- | --- |
| Fail | Mapping relies on surface similarity or omits required relations. |
| Pass | Required relations are preserved and irrelevant similarities are ignored. |
| Excellent | Mapping preserves required relations and explains where the mapping stops. |

### Open-Task Audit Rubric

| Rating | Observable Meaning |
| --- | --- |
| Fail | Critical errors missed, uncertainty hidden, or corrections made without evidence. |
| Pass | Critical errors found, uncertainty marked, and corrections justified. |
| Excellent | Critical errors found, noncritical errors mostly found, uncertainty well bounded. |
